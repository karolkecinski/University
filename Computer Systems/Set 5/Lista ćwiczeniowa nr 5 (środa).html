Zadanie 1

Piotr Witkowski Zadanie rozwiązuje p. Przemysław Czerwiński.

Zdefiniuj pojęcie wyrównania danychwyrównania danych w pamięci (ang. \emph{data alignment}). Dlaczego dane typów prostych (np. shortshort, intint, doubledouble) powinny być w pamięci odpowiednio wyrównane? Jaki związek z wyrównywaniem danych mają wypełnieniawypełnienia (ang. \emph{padding}) w danych typu strukturalnego. Odpowiadając na powyższe pytanie podaj przykład struktury, której rozmiar w bajtach (wyliczony przez operator sizeofsizeof) jest większy niż suma rozmiaru pól składowych. Czemu służy wypełnienie wewnętrznewypełnienie wewnętrzne (ang. \emph{internal padding}) a czemu wypełnienie zewnętrznewypełnienie zewnętrzne (ang. \emph{external padding})?

Rozwiązanie

Wyrównanie danych w pamięci polega na ustawieniu danych tak, aby zaczynały się na indeksie będącym wielokrotnością pewnej liczby (w przypadku typów prostych jest to wielokrotność rozmiaru typu – np. 4 w przypadku int32_t).
Jest to potrzebne, bo procesor jest w stanie szybciej pracować w takich warunkach, niż gdyby dane mogły zaczynać się gdziekolwiek.

Wypełnienie danych w structach ma właśnie na celu wyrównanie danych, aby zaczynały się na odpowiednich wielokrotnościach.
Mając na przykład structa składającego się ze zmiennej typu char (1 bajt) oraz int32_t (4 bajty), po charze nastąpiłyby 3 puste bajty, aby int32_t zaczynał się na indeksie będącym wielokrotnością 4.
Jest to przykład wypełnienia wewnętrznego, a także przykład structa, którego rozmiar jest większy niż suma rozmiarów składowych – 8 zamiast 1+4.

Wypełnienie zewnętrzne służy temu, aby rozmiar całego structa był odpowiedni – na koniec structa dokłada się tyle pustych bajtów, aż rozmiar structa będzie wielokrotnością rozmiaru największego typu w tym structcie – czyli na przykład jeśli największy typ w danym structcie to int32_t, wówczas rozmiar structa musi być wielokrotnością 4.

Rozwiązanie

Spójrzmy do wspomnianego w treści dokumentu:

(Definicje)

INTEGER This class consists of integral types that fit into one of the general purpose registers.
(…)
MEMORY This class consists of types that will be passed and returned in memory via the stack.

The classification of aggregate (structures and arrays) and union types works as follows:

  1. If the size of an object is larger than eight eightbytes, or it contains unaligned fields, it has class MEMORY.
  2. If the size of the aggregate exceeds a single eightbyte, each is classified separately. Each eightbyte gets initialized to class NO_CLASS.
  3. Each field of an object is classified recursively so that always two fields are considered. The resulting class is calculated according to the classes of the fields in the eightbyte:
    (a) If both classes are equal, this is the resulting class.
    (b) If one of the classes is NO_CLASS, the resulting class is the other
    class.
  4. Then a post merger cleanup is done:
    (a) If one of the classes is MEMORY, the whole argument is passed in memory.

W oparciu o powyższą analizę przypadków, zostaje podjęta decyzja o sposobie przekazania argumentów:

  1. If the class is MEMORY, pass the argument on the stack.
  2. If the class is INTEGER, the next available register of the sequence %rdi, %rsi, %rdx, %rcx, %r8 and %r9 is used.

Konsekwencje punktu drugiego widzieliśmy już w wielu zadaniach i rozpoznajemy opisaną w nim kolejność użycia rejestrów.

Podsumowując powyższe fragmenty: duże struktury, takie które mają rozmiar większy niż 64 bajty, są umieszczane na stosie. W przeciwnym razie są przekazywane przez rejestry w znanej nam kolejności.

By lepiej zrozumieć powyższe reguły i przyjrzeć się ich konsekwencjom dla różnych struktur, przygotowałem program realizujący różne sposoby użycia struktur:





























typedef struct first T; // łatwiejsze zmienianie struktur

T returns() {
    T x;
    x.val1 = 1;
    return x;
}

void caller() {
    T x = returns();
}

void updates(T x) {
    x.val1 = 1;
}

void caller1() {
    T x;
    updates(x);
}

void rec(T x, int n) {
    if (n == 0) {
        x.val1 = 1;
    } else {
        rec(x, n-1);
    }
}

Zadanie 8

Piotr Witkowski Zadanie rozwiązuje p. Karol Kęciński.

Język CC dopuszcza deklaracje tablic wielowymiarowych zopuszczonym rozmiarem pierwszego wymiaru. Taka deklaracja może wystąpić w nagłówku funkcji, np. \Symbol{void process(int32_t A[][77], size_t len)}. Nie można natomiast opuszczać rozmiarów innych wymiarów, np. \Symbol{void bad(int32_t A[77][], size_t len)} nie jest poprawną deklaracją. Wyjaśnij, dlaczego tak jest odwołując się do sposobu, w jaki kompilator tłumaczy odwołania do tablic z CC na asembler.

Rozwiązanie:

Niech A[Wym1][Wym2] to nasza tablica.

Załóżmy, że znamy Wym2. Wtedy znamy “rozmiar każdego z segmentów A[i][Wym2]” (Wiemy o ile bajtów trzeba przesunąć wskaźnik z A[i][Wym2] do A[i+1][Wym2]).
Analogicznie w przypadku A[i][Wym2][Wym3]…

Natomiast znając pierwszy wymiar A[Wym1][j], wiemy jedynie “ile” mamy elementów, lecz nie znamy ich długości (analogicznie w przypadku A[Wym1][Wym2][k]). Z tego powodu nie wiadomo o ile należy przesuwać wskaźnik.
